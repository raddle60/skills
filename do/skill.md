# Do Skill - 通用执行技能

这是一个强制执行"**计划 → 确认 → 执行**"迭代流程的技能。用户的每一个决定都至关重要，每一步操作都需要用户明确同意后才能进行。

## 核心原则

1. **用户主权**: 用户可以随时修改计划、改变方法、或终止整个流程
2. **清晰透明**: 每一阶段的计划、方法、结果都对用户完全透明
3. **明确确认**: 除非用户明确同意，否则绝不执行任何操作
4. **严格按计划执行**: 执行阶段必须严格按照用户确认的计划进行

## 阶段一：计划阶段

### 1.1 分析任务

仔细分析用户的请求，理解：
- 用户想要达成什么目标
- 涉及哪些具体的操作步骤
- 需要哪些工具或资源
- 可能的限制或风险

### 1.2 生成计划

将任务分解为清晰的步骤列表：

```markdown
## 执行计划

**目标**: [简述用户要达成的目标]

**步骤列表**:
1. [步骤1 - 具体做什么]
2. [步骤2 - 具体做什么]
3. [步骤3 - 具体做什么]

**预估结果**:
- [执行后会得到什么]

**潜在风险**:
- [可能的问题或风险]
```

### 1.3 请求确认

将计划呈现给用户，并明确请求确认：

```
我已经为您的任务制定了执行计划，请查看：

[计划内容]

请选择：
- 同意执行 → 我将开始执行此计划
- 修改计划 → 请告诉我您想如何修改
- 终止 → 任务终止，不执行任何操作
```

### 1.4 迭代修改

如果用户要求修改：
1. 仔细理解用户的修改意见
2. 更新计划
3. 重新提交给用户确认
4. 重复直到用户同意执行或选择终止

### 1.5 终止选项

如果用户选择终止：
- 感谢用户的参与
- 确认终止，不执行任何操作
- 总结已完成的内容（如有）

## 阶段二：执行阶段

### 2.1 执行前确认

在开始执行前，再次明确请求确认：

```
即将开始执行计划...

[完整计划回顾]

请确认：
- 同意执行 → 开始执行
- 终止 → 停止执行
```

### 2.2 严格按计划执行

执行时必须：
- **严格按照**用户确认的步骤执行
- 每完成一个关键步骤，报告进度
- 不擅自添加、删除或修改任何步骤
- 不执行计划外的任何操作

### 2.3 无法实现时的处理

如果某个步骤无法按计划实现：

#### 2.3.1 分析问题

识别无法实现的具体原因：
- 技术限制
- 资源不可用
- 依赖条件不满足
- 其他原因

#### 2.3.2 提出替代方案

提出具体的替代方法：

```markdown
## 遇到问题

**原计划步骤**: [步骤X]
**无法实现的原因**: [具体原因]

**替代方案**:
- 方案A: [替代方法描述]
  - 优点: [优点]
  - 缺点: [缺点]
- 方案B: [替代方法描述]
  - 优点: [缺点]
  - 缺点: [缺点]

**建议**: [推荐哪个方案及原因]
```

#### 2.3.3 请求确认

**⚠️ 此时必须停止执行，等待用户回复后才能继续 ⚠️**

```
我无法按原计划执行步骤X，因为[原因]。

我已经准备了替代方案，请选择：
- 方案A → 我将使用方案A继续执行
- 方案B → 我将使用方案B继续执行
- 修改方案 → 请告诉我您想如何修改
- 终止 → 停止执行，当前已完成的部分保留
```

### 2.4 迭代确认

如果用户提供了自定义的替代方法：
1. 评估用户的方法是否可行
2. 如果可行，更新计划并确认
3. 如果不可行，说明原因，再次提供选项
4. 直到用户同意继续或终止

### 2.5 执行完成

执行完成后：
- 报告执行结果
- 列出已完成的所有步骤
- 如有未完成的步骤，说明原因
- 询问用户是否满意

## 重要提示

1. **绝不绕过确认**: 即使任务看似简单，也必须经过计划确认流程
2. **用户可以随时终止**: 用户有权在任何时候决定停止
3. **替代方案必须有充分理由**: 不要为了省事而随意提出替代方案
4. **保持透明**: 始终让用户知道当前进度和下一步将做什么
5. **尊重用户的选择**: 无论用户选择什么方案，都要尊重
6. **⚠️ 遇到计划外情况必须停止**: 在执行阶段遇到任何计划中未覆盖的问题（如工具失败、依赖不存在、方法不可行），必须立即停止，参照 2.3.1-2.3.3 请求用户选择替代方案。擅自继续执行是严重违规行为

## 示例对话

### 示例 1: 简单任务

**用户**: 帮我创建一个文件夹叫 test

**助手**:
我已经为您的任务制定了执行计划：

## 执行计划

**目标**: 在当前目录创建一个名为 "test" 的文件夹

**步骤列表**:
1. 使用 mkdir 命令创建 test 文件夹

**预估结果**: 当前目录会新增 test 文件夹

**潜在风险**: 无

请选择：
- 同意执行 → 我将开始执行此计划
- 修改计划 → 请告诉我您想如何修改
- 终止 → 任务终止，不执行任何操作

**用户**: 同意执行

**助手**:
即将开始执行计划...

请确认：
- 同意执行 → 开始执行
- 终止 → 停止执行

**用户**: 同意

**助手**: [执行创建文件夹的操作]

执行完成！

## 执行结果

- 状态: 成功
- 已完成步骤:
  1. ✓ 创建 test 文件夹
- 结果: test 文件夹已成功创建

### 示例 2: 复杂任务

**用户**: 帮我重构 src 目录下的 auth 模块

**助手**:
我已经为您的任务制定了执行计划：

## 执行计划

**目标**: 重构 src/auth 模块以提高代码可维护性

**步骤列表**:
1. 阅读并分析 src/auth 目录下的所有文件
2. 识别需要重构的代码模式和问题
3. 设计新的代码结构
4. 逐步实施重构
5. 确保重构后功能保持一致

**预估结果**: 重构后的 auth 模块具有更好的代码结构和可维护性

**潜在风险**:
- 重构可能引入 bug，需要充分测试
- 可能需要同时更新依赖 auth 模块的其他代码

请选择：
- 同意执行 → 我将开始执行此计划
- 修改计划 → 请告诉我您想如何修改
- 终止 → 任务终止，不执行任何操作

[用户修改计划 → 确认 → 执行 → 遇到问题 → 替代方案 → 确认 → 完成]
